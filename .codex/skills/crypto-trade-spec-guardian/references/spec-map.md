# Карта спек CryptoTrade

Используй эту карту, чтобы выбрать минимальный набор спек для конкретной задачи.

## Главный ориентир

- `docs/Spec/1 Functional.md`: главный рабочий спек проекта; основные разделы сайта, поведение фич и главные пользовательские сценарии.
- `docs/Spec/Global.spec.md`: смысл продукта, аудитория, назначение и границы проекта. Читать только когда нужно проверить общий смысл, аудиторию или рамки продукта.

## Примеры выбора спек

- Задача про UI топ монет: читать `docs/Spec/1 Functional.md`, `docs/Spec/feature-specs/топ-монеты.md`, `docs/Spec/feature-specs/состояния-и-ошибки.md`, `docs/Spec/feature-specs/мобильная-версия.md` и связанные спеки по данным/API.
- Задача про полноэкранный график: читать `docs/Spec/1 Functional.md`, `docs/Spec/feature-specs/спотовый-график.md`, `docs/Spec/technical-specs/графики-и-tradingview.md` и мобильную спеку, если меняется layout.
- Задача про сохранение после авторизации: читать `docs/Spec/1 Functional.md`, `docs/Spec/feature-specs/регистрация-и-вход.md`, `docs/Spec/feature-specs/сохранение-данных.md`, `docs/Spec/technical-specs/авторизация-и-хранение.md`.
- Задача про ревью фичи: читать `docs/Spec/1 Functional.md`, спеку нужной фичи, связанную техническую спеку и user story, если меняется пользовательский путь.
- Задача про user story: читать `docs/Spec/1 Functional.md`, релевантные feature-specs и формулировать критерии приемки наблюдаемым языком.

## Когда нужно подсветить неясность

Перед кодингом обязательно отмечай проблему, если:

- Фича есть в коде, но ее нет в спеках.
- В спеке написано `может`, но для реализации нужно точное правило.
- Спека говорит о сохранении данных, но не разделяет поведение гостя и авторизованного пользователя.
- Спека говорит о недоступных данных, но не определяет fallback UI.
- Описано поведение для desktop, но не описано поведение для mobile.
- Используется статус, цвет, источник, API-поле или правило хранения без явной опоры на спеку.
- Техническая спека и спека фичи подразумевают разное поведение.
