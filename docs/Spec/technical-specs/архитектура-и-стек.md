# Technical spec: Архитектура и стек

## 1. Назначение

Эта спека фиксирует техническую основу проекта CryptoTrade.

Она нужна, чтобы разработка не начиналась с открытых вопросов:
- на чем писать frontend;
- нужен ли backend;
- где хранить пользователей;
- где хранить халяль-данные;
- как разделить UI, данные и бизнес-логику.

## 2. Выбранный стек для MVP

Frontend:
- React;
- TypeScript;
- Next.js;
- маршрутизация через возможности Next.js;
- CSS Modules, Tailwind CSS или другой единый styling-подход.

Backend:
- для быстрого MVP можно использовать BaaS;
- если нужен свой backend, использовать Node.js + NestJS или Express/Fastify.

База данных:
- PostgreSQL для аккаунтов, прогресса, избранного и халяль-статусов;
- localStorage или IndexedDB для гостевых настроек.

Авторизация:
- email/password;
- сессии через выбранный backend или BaaS;
- refresh/session token не должен храниться в небезопасном виде, если используется свой backend.

## 3. Архитектурные слои

Проект должен быть разделен на слои:
- UI components;
- pages/routes;
- feature modules;
- API clients;
- domain logic;
- storage adapters;
- shared types.

Логика халяль-проверки не должна быть зашита прямо в UI-компоненты.

Логика получения цен и графиков не должна быть смешана с визуальной отрисовкой таблиц и графиков.

## 4. Основные модули

Минимальные модули:
- `auth`;
- `coins`;
- `halalCheck`;
- `charts`;
- `learning`;
- `watchlist`;
- `settings`;
- `i18n`;
- `layout/navigation`.

## 5. Что надо определить глубже

Нужно принять решение:
- BaaS или собственный backend;
- какую базу данных использовать;
- будет ли админская часть для обновления халяль-статусов;
- где хранить обучающий контент;
- будет ли серверный кэш для внешних API.

## 6. Критерии готовности

Спека считается закрытой, когда:
- стек Next.js учтен в структуре проекта;
- описана структура папок проекта;
- выбран способ авторизации;
- выбран способ хранения данных;
- понятно, где будет бизнес-логика.
