# Technical spec: Халяль-проверка

## 1. Назначение

Эта спека описывает техническую модель предварительной оценки монеты по халяль-критериям.

Проект не должен выдавать религиозное заключение. Он показывает образовательную предварительную оценку, объяснение и источники.

## 2. Статусы

Разрешенные статусы:
- `Halal-friendly`;
- `Not halal`;
- `Unknown`.

Эти статусы должны совпадать с feature-спеками и functional-спекой.

## 3. Цвета статусов

Цвета должны быть едиными во всем проекте:
- `Halal-friendly`: зеленый;
- `Not halal`: красный;
- `Unknown`: серый.

UI не должен назначать цвет отдельно от статуса вручную. Цвет должен приходить из единой status-map.

## 4. Модель результата

Результат проверки должен содержать:
- `coinId`;
- `status`;
- `confidence`;
- `summary`;
- `risks`;
- `criteria`;
- `sources`;
- `lastReviewedAt`;
- `reviewedBy`;
- `disclaimer`.

`confidence` не должен создавать иллюзию фатвы. Он показывает техническую уверенность в данных, а не религиозную окончательность.

## 5. Критерии проверки

Критерии нужно описать отдельно и поддерживать как данные, а не только как текст в UI.

Минимальные группы критериев:
- назначение проекта;
- наличие явно харам-деятельности;
- механизм доходности;
- staking/lending/interest-механики;
- gambling/high-risk utility;
- прозрачность команды и документации;
- доступность источников;
- противоречивые сигналы.

## 6. Источники

Для каждого решения нужно хранить источники:
- официальный сайт проекта;
- whitepaper или документация;
- публичные страницы о токеномике;
- внешние аналитические источники;
- дата проверки.

Если источников недостаточно, статус должен быть `Unknown`, а не уверенный `Halal-friendly`.

## 7. Что надо определить глубже

Нужно подробно определить:
- точные критерии;
- вес критериев;
- кто обновляет халяль-статусы;
- как хранить историю изменений;
- как показывать пользователю причины решения;
- какие источники считаются достаточными.
